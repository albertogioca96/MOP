# Iteration Log ‚Äî MVP Build

## What we built

### Session 1 ‚Äî 2026-04-17 (full app from scratch)

Started from zero. Scaffolded the Vite + React + TypeScript app and built the entire MVP in one session.

**Core architecture:**
- Single React Context store (`StoreContext` + `buildStore`) ‚Äî no Redux, no Zustand, just hooks
- `entries: Record<string, Entry>` keyed by `date__slot__consultantId` ‚Äî the central data structure everything reads from
- Pure engine functions (allocate, capacity, budget, history) fully decoupled from React

**What got built in order:**
1. Types + seed data (3 consultants, 3 clients)
2. Store with all state and actions
3. Engine: allocate (greedy draft), capacity warnings, budget calc, history snapshots
4. NavBar with month tracker (12 pills + year navigation)
5. Planner grid ‚Äî full table, sticky headers, AM/PM rows, sync scrollbar, weekend narrow columns
6. Planner cell ‚Äî filled/empty/weekend states, compact mode for full-month view
7. Brush tool ‚Äî assign/erase modes, team propagation on click, trash cursor for erase mode
8. Day labels (Holiday/Strategy/Event) ‚Äî visual column tinting, never blocks writing
9. Log view ‚Äî date-grouped table, inline edit, focus mode from cell click
10. Dashboard ‚Äî utilization bars + budget summary per client
11. Setup view ‚Äî client cards + consultant cards, editable inline

**UI decisions made:**
- Dark theme throughout (#0f172a / #1e293b / #334155 palette)
- No dropdowns per cell ‚Äî click = action, popup = detail only
- Compact "All" toggle shows full month with client name only (no icons)
- Real scrollbar above header (synced) instead of nav buttons
- Narrow Sat/Sun columns (34px) to save space

**Bugs fixed during build:**
- Unused TS imports causing build errors (LogView)
- `?? unreachable` TS2869 in PlannerCell color expression
- `buildStore` not exported
- `setVisibleDays` missing from store return object
- `onCancelNew` for new clients left ghost entries ‚Äî fixed by adding `handleDeleteClient`

## What worked well

- The store pattern (single `buildStore` function with all hooks) scaled well ‚Äî easy to add state without refactoring
- Pure engine functions made the conflict detection trivial to add without touching React
- The `entries` key format (`date__slot__consultantId`) made all queries O(1) lookups
- Inline edit pattern in cards (draft state, save/cancel) was clean and reusable

## What to improve next

- **Data persistence** ‚Äî localStorage or IndexedDB so the app survives a refresh
- **Budget-maximizing allocation** ‚Äî after 1-day-per-week baseline, fill remaining capacity with extra days for high-priority clients
- **Export** ‚Äî PDF or Excel of the monthly plan
- **Multi-month view** ‚Äî carry forward context across months
- **Manual priority override** ‚Äî drag to reorder clients in the priority list
